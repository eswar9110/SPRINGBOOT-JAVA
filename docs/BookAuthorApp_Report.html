1. How I Designed the Data Model

I picked Authors and Books because the relationship between them is intuitive and maps well to a relational schema. Every writer can pen multiple titles, but each title is attributed to a single writer — making this a textbook one-to-many association.

✍ Author Table

id — Primary Key (auto-incremented)

name — Writer's full name (mandatory)

email — Contact address (must be unique)

bio — Short biography (up to 500 chars)

1 ───→ N

📖 Book Table

id — Primary Key (auto-incremented)

title — Name of the publication (mandatory)

isbn — International identifier (must be unique)

price — Retail cost in USD (mandatory)

year_published — When it hit the shelves

author_id — References the writer who created it

Why this design? The foreign key author_id in the Book table points back to the Author's primary key. I annotated the Author side with @OneToMany(mappedBy="author") and the Book side with @ManyToOne combined with @JoinColumn. Cascading is set to ALL so that when I persist an author, any associated books get persisted too.

3. Building the Application Layer by Layer

3.1 Defining the Domain Objects

My two domain classes carry JPA annotations that tell Hibernate how to map them to database tables. I also added Bean Validation constraints so the framework rejects bad input before it ever reaches the database.

Author — the "one" side

@Entity @Table(name = "authors")
public class Author {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @NotBlank @Size(min = 2, max = 100)
    @Column(nullable = false)
    private String name;

    @NotBlank @Email @Column(nullable = false, unique = true)
    private String email;

    @Size(max = 500) @Column(length = 500)
    private String bio;

    @OneToMany(mappedBy = "author", cascade = CascadeType.ALL, fetch = FetchType.LAZY)
    private List<Book> books = new ArrayList<>();
}

Book — the "many" side

@Entity @Table(name = "books")
public class Book {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @NotBlank @Column(nullable = false)
    private String title;

    @NotBlank @Column(nullable = false, unique = true)
    private String isbn;

    @NotNull @DecimalMin("0.0") @Column(nullable = false)
    private Double price;

    @Min(1000) @Max(2030)
    @Column(name = "year_published")
    private Integer yearPublished;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "author_id", nullable = false)
    private Author author;
}

3.2 Writing the Data Access Logic

Spring Data JPA eliminates the need for manual SQL in most cases. I simply declared interfaces that extend JpaRepository, and Spring auto-generated the implementation at startup.

Author data access

@Repository
public interface AuthorRepository extends JpaRepository<Author, Long> {
    Author findByEmail(String email);  // derived query from method name
}

Book data access — with a hand-written JPQL join

@Repository
public interface BookRepository extends JpaRepository<Book, Long> {

    @Query("SELECT new com.bookauthor.app.dto.BookAuthorDTO("
         + "b.title, b.isbn, b.price, b.yearPublished, a.name, a.email) "
         + "FROM Book b INNER JOIN b.author a")
    List<BookAuthorDTO> findAllBooksWithAuthors();

    Book findByIsbn(String isbn);
}
About the join query: Rather than pulling separate lists and merging them in Java, I wrote a single JPQL statement that joins both tables on the foreign key. The NEW keyword lets me project the result directly into a lightweight DTO, avoiding unnecessary entity overhead.

3.3 Implementing Business Rules

My service classes sit between the controllers and repositories. They encapsulate the workflow for each operation and translate database exceptions into meaningful feedback.

@Service
public class AuthorService {
    private final AuthorRepository repo;

    public List<Author> findAll()          { return repo.findAll(); }
    public Optional<Author> findById(Long id) { return repo.findById(id); }

    public Author save(Author a) throws DataIntegrityViolationException {
        return repo.save(a);   // exception if email is a duplicate
    }

    public Author update(Long id, Author incoming) {
        Author existing = repo.findById(id)
            .orElseThrow(() -> new RuntimeException("No author with id " + id));
        existing.setName(incoming.getName());
        existing.setEmail(incoming.getEmail());
        existing.setBio(incoming.getBio());
        return repo.save(existing);
    }
}

The BookService mirrors this pattern and adds a getBooksWithAuthors() method that delegates to the custom repository query.

3.4 Handling Web Requests

Each controller is annotated with @Controller and uses @GetMapping / @PostMapping to route HTTP traffic. I rely on @Valid for server-side validation and catch DataIntegrityViolationException to show friendly error messages when a unique constraint is violated.

@Controller @RequestMapping("/authors")
public class AuthorController {
    @GetMapping              // renders authors/list.jsp with all records
    @GetMapping("/add")      // renders the empty form
    @PostMapping("/add")     // validates, saves, catches duplicates
    @GetMapping("/edit/{id}")  // loads record into edit form
    @PostMapping("/edit/{id}") // validates, updates, catches duplicates
}

4.2 Adding New Entries

The "Add" forms let users type in fresh data. On submission, the controller checks validation annotations first. If everything passes, it hands off to the service layer. If the database rejects the insert (say, a duplicate email or ISBN), the catch block redisplays the form with a warning banner.

7. Difficulties I Ran Into and My Workarounds

What Went WrongHow I Fixed It
My machine only had the Java Runtime — the compiler (javac) wasn't available anywhere in PATHI discovered an OpenJDK 8 build tucked inside the Android SDK installation folder. I pointed JAVA_HOME there and it compiled everything without issues.
Maven wasn't installed globallyDownloaded a portable Maven 3.8.8 distribution into my user profile and added its bin/ to PATH for the session. No admin rights needed.
One JSP file silently failed to save — the Add Book page was returning a 404 at runtimeTraced it to a file-write that was interrupted mid-process. I recreated books/add.jsp from scratch and restarted the embedded Tomcat.
Binding the author dropdown in the Book form — Spring Form tags don't natively support foreign-key dropdownsI used a plain HTML <select> populated with JSTL, then captured the selection via @RequestParam("authorId") in the controller.
Unique constraint violations crashed the form — entering a duplicate email or ISBN threw an unhandled exceptionWrapped the save() calls in try-catch blocks for DataIntegrityViolationException and redirected back to the form with a user-readable message.
Lazy-loaded author names in the Book list threw exceptionsSpring Boot's default open-in-view setting keeps the Hibernate session alive during JSP rendering, so lazy fields resolve automatically. I kept this default enabled.
