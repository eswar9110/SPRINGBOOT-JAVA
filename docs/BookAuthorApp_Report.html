1. Schema Blueprint

I modelled two real-world concepts: literary writers and the publications they produce. A single writer may have penned dozens of titles, but every title traces back to exactly one creator. This naturally maps to a parent-child table structure connected by a foreign key.

Writer Table

id — auto-generated numeric key

name — full name, cannot be blank

email — contact info, enforced unique

bio — optional blurb, capped at 500 characters

1 ——→ N

Publication Table

id — auto-generated numeric key

title — name of the work, cannot be blank

isbn — globally unique book identifier

price — retail cost, must be positive

year_published — four-digit year

author_id — points to the writer who created it

Mapping strategy: The parent entity carries a collection reference marked with the "mapped-by" attribute, while the child entity holds the actual foreign-key column. I enabled cascading so changes to a writer automatically ripple to associated publications.

3. Constructing Each Software Layer

Domain Models

Both classes are annotated so Hibernate knows which table they correspond to and how columns should be constrained. Validation annotations on each field tell the framework to reject malformed input at the controller boundary — before any database call occurs.

On the Writer side, I declared the primary key with an identity generation strategy, marked the email column as unique, and added a lazily-fetched collection that represents all publications by that writer.

On the Publication side, the ISBN carries a uniqueness constraint, the price field requires a non-negative value, and a many-to-one association links each row back to its parent writer through a foreign-key column called author_id.

Data Interfaces

Spring Data lets me define an interface rather than a concrete class. At startup the framework fabricates a full implementation that includes methods for persisting, fetching by primary key, listing all rows, and more.

For the Writer interface, I added one extra method: findByEmail — Spring figures out the correct SQL from the method name alone.

For the Publication interface, I went further and wrote a JPQL statement by hand. It performs a cross-table join between publications and writers, then packs each result row into a lightweight transfer object. This avoids pulling entire entity graphs into memory when all I need is a flat table of combined fields.

-- Pseudocode of the join query:
SELECT publication.title, publication.isbn, publication.price,
       publication.year_published, writer.name, writer.email
FROM   publications  INNER JOIN  writers
ON     publications.author_id = writers.id
Why a DTO? Returning full entity objects from a join would trigger unnecessary lazy-loading chains and expose internal structure to the view layer. The transfer object contains only the six fields I actually display, keeping things lean.

Middleware Logic (Service Classes)

Each service class acts as a gatekeeper between the web layer and the database. The core responsibilities are:

  • Listing — delegate to the repository's built-in findAll
  • Lookup — delegate to findById, returning an Optional wrapper
  • Creation — call save and let any constraint-violation exceptions propagate upward
  • Modification — fetch the existing record first, overwrite its fields with the incoming values, then persist it again. If the ID doesn't match anything, throw a runtime exception.
  • Cross-table query (publications only) — forward to the custom repository method

Request Routing (Controllers)

I created two primary controllers, each scoped to a URL prefix. Every handler method either returns a view name (for GET requests) or redirects after processing a form (for POST requests). Flash attributes carry success or error banners across the redirect.

Key design choices in the controllers:

  • The @Valid annotation triggers bean-validation before my code runs
  • A BindingResult parameter lets me detect validation failures and re-render the form
  • A try-catch around save/update catches uniqueness violations and shows a human-friendly warning
  • The publication form receives the writer's ID as a separate request parameter, which I resolve into a full entity before attaching it to the publication

Front-End Templates

Seven JSP files power the interface. Each one follows the same structural pattern: a navigation strip at the top, a content area in the middle, and conditional banners for success or error feedback. I used JSTL's iteration tag to loop over collections and Spring's form tags to bind input fields directly to model attributes.

Visually, I went for a dark gradient backdrop with semi-transparent card panels that float above it. Prices render in green, writer names appear inside coloured pills, and buttons subtly shift upward on hover. All styling is hand-written — no CSS framework involved.

4. Demonstrating the Three Core Workflows

Seeding the Database

When the application boots, Hibernate drops and recreates both tables. Immediately after, Spring executes a SQL script I placed in the resources folder. It inserts ten well-known literary figures (Rowling, Orwell, Austen, Twain, Tolstoy, Christie, Hemingway, Marquez, Dostoevsky, Lee) and one signature title from each.

Adding Fresh Records

Navigating to the creation form presents empty input fields. For publications, a dropdown menu lists every writer currently in the database — populated dynamically through JSTL. On submission, the server validates all constraints. If anything violates a uniqueness rule, the user sees a coloured warning banner instead of a raw stack trace.

The listing pages render every record in a styled table. The combined view is the most technically interesting: it runs the custom JPQL join and renders a merged table showing publication details alongside the associated writer's name and email. 6. Obstacles Encountered During Development

ProblemResolution
The development machine had only a runtime environment — no compiler binary was on the system pathLocated a compatible compiler bundled inside the Android SDK directory and configured the environment variable to point there
The build tool was absent from the system entirelyFetched a self-contained Maven distribution and loaded it into the session path — no system-wide installation required
One template file was silently not created during the initial scaffolding phase, causing a 404 at runtimeIdentified the missing file through server logs, recreated it manually, and restarted the embedded container
Linking a child record to its parent through a form dropdown required a workaround since the tag library does not natively handle foreign-key selectionsUsed a plain HTML select element populated through iteration tags, and captured the chosen value as a separate request parameter in the handler method
Submitting a form with a value that already existed in a unique column resulted in an unhandled exception pageWrapped persistence calls in exception handlers that intercept constraint violations and re-display the form with a descriptive warning
Accessing a lazily-fetched association inside a template triggered a session-closed error in some configurationsRelied on the framework's default behaviour of keeping the persistence context open throughout view rendering
